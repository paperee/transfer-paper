## 囫囵吞枣与速成后的回过头重修

> Date: 2026.05.10

### 当我想速成一门编程语言…

我使用的方法是：挑选几篇该语言的代码，把它们丢给 AI，让 AI 逐字逐句讲解这些代码中使用的语法与特性。

这是一种 `应用型` 的学习方法，我拿到的是能立马用得上的工具（即该语言的语法与特性）

如果你事先学过其他编程语言，并且能把类比与对比应用到这门新语言与旧语言上，那是再高效不过了。

事后，我让 `claude-haiku-4.5` 评估这种学习方法，它认为，这种 `自上而下` 的方法有如下优势：

- **可用性快速**：_能够迅速达到"写出能跑的代码"这个里程碑，在心理上产生正反馈，激励继续学习；_
- **上下文驱动学习**：_具体的应用场景能让大脑建立"功能导向"的认知网络，而不是孤立的语法规则；_
- **语法衔接更紧凑**：_实际代码的分析易于帮助理解语法的用法，而非死记硬背"定义变量"等抽象规则。_

但同时，也存在几个隐藏的风险与盲区：

- **样本偏差问题**：_几篇样本代码可能不够有代表性，学会的只会是该语言的某个特定风格，而不是该语言本身；_
- **隐性假设和边界情况**：_完整代码通常在"常见路径"上运行良好，但边界情况往往不会在简洁的样本代码中出现；_
- **性能和可扩展性盲点**：_"能用"的100行代码不一定"高效"，样本代码中的性能问题或不可扩展的设计可能被搬运；_
- **错误调试能力的缺陷**：_当代码出错时，可能搞不清底层原因。不理解底层机制旧只能通过低效的"试错"来解决问题。_

因而，在快速入门后，我决定 `自下而上` 地学习《[Haskell 趣学指南](https://flaneur2020.github.io/)》

#### 惰性求值与 n 阶单位矩阵

尝试利用递归与 Haskell 的惰性求值特性实现该功能，以下是思路（伪代码）：

```haskell
ghci> -- mtx n = [[1*n]*n]
ghci> -- rep ? = ?:rep ?
ghci> -- -> rep 1 = 1:rep 1
ghci> -- -> rep [1,1,…] = [1,1,…]:rep [1,1,…]
ghci> -- mtx n -> take n rep (take n rep 1)
```

简洁的函数，n 表示 n 阶单位矩阵：

```haskell
mtx :: Int -> [[Int]]
mtx n = take n (rep (take n (rep 1))) -- take 函数用于取无限列表的前 n 项
  where rep x = x:(rep x) -- 将 x 单独提出来而不是包在 rep 中，目的是记录递归节点
```

## 编译时的报错截断与终端编码问题

> Date: 2026.05.05

### 编译时的报错截断

当我首次尝试使用 `stack built` 编译时，控制台抛出如下报错：

```
Error: [S-7282]
       Stack failed to execute the build plan.
       While executing the build plan, Stack encountered the error:
       <stderr>: commitAndReleaseBuffer: invalid argument (cannot encode character '\65533')
```

报错表示，错误输出 (stderr) 无法编码 `\65533` 字符。

在 Unicode 中，`\65533` 对应 `U+FFFD` 替换字符 (Replacement Character)。

它不是一个有意义的实际字符，而是用来表示数据流中无法被识别、被解码、被正确表示的字符。

也就是说，错误输出中存在不兼容的字符，终端不认识它们，就统一把它们当作 `\65533`。

使用 `chcp` 查看当前终端的编码，输出 `活动代码页: 936`，表示当前终端是 GBK (936) 编码。

使用 `chcp 65001` 切换到 UTF-8 (65001) 编码，重新编译，正常输出，并显示了更详细的 GHC 错误信息：

```
app\Main.hs:37:7: error: [GHC-83865]
    鈥?Couldn't match type 鈥楨ither RestCallErrorCode ()鈥?with 鈥?)鈥?[0m
    ……
      In a stmt of a 'do' block:
        when (self `isInfixOf` messageContent msg)
          $ do restCall
                 $ CreateReaction (messageChannelId msg, messageId msg) "fire"
   |
37 |       restCall $ CreateReaction (messageChannelId msg, messageId msg) "fire"
   |       ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
```

第一行后面出现 `[0m`，它是一个用来重置所有文本属性的 ANSI 转义序列，通常用于实现彩色输出。

而第一行的 `鈥` 和 `楨` 等乱码，应该是颜色码中某些无法在当前编码下被正确编码的字节组合。

推测 GHC 在输出彩色错误信息时，生成了包含 ANSI 转义码的序列，它们在 GBK 编码下无法被正确解释。

在当前 `65001` 代码页上运行编译后的可执行文件，却发现输出的所有中文都变成了乱码。

尝试使用 `chcp 936` 切换回 GBK 编码，重新运行可执行文件，输出的中文正常。

为什么在 UTF-8 环境下编译错误能正常显示，但运行时的中文输出又需要在 GBK 环境下才正常显示呢？

也许 Haskell 程序在输出文本时，默认编码并非 UTF-8，或者存在某种混合编码的处理机制。

emmm… 这样一来，编译和执行必须在两个代码页来回切换，十分麻烦。

### 终端编码问题

这里有三条路可以走：一、禁用 GHC 的彩色输出；二、强制 GHC 用 JSON 输出；三、强制程序用 UTF-8 输出。

#### 一、禁用 GHC 的彩色输出

我尝试在 `ghc-options` 中添加 `-fdiagnostics-color=never`，重新编译，输出：

```
C:\sr\setup-exe-cache\x86_64-windows\Cabal-simple_O_vy6YIf_3.12.1.0_ghc-9.10.3.exe --verbose=1 --builddir=.stack-work\dist\1a191874 build lib:hcc-eebot exe:hcc-eebot-exe --ghc-options " -fdiagnostics-color=always"
```

奇怪的是，编译时输出的路径和参数表明，实际执行的编译命令中携带的参数却是 `-fdiagnostics-color=always`。

自定义的选项似乎被优先级更高的配置覆盖，或者存在冲突。

尝试使用 `stack build --no-interleaved-output` 禁用 GHC 的交错输出模式，同样没用。

#### 二、强制 GHC 用 JSON 输出

我尝试在 `ghc-options` 中添加 `-fdiagnostics-as-json`，重新编译，输出：

```
{"version":"1.0","ghcVersion":"ghc-9.10.3","span":{"file":"app\\Main.hs","start":{"line":122,"column":87},"end":{"line":122,"column":88}},"severity":"Error","code":58481,"message":["parse error on input `)'"],"hints":[]}
```

JSON 是一种结构化的文本格式，不依赖于终端的特定编码来表示特殊字符。

如上，行号与错误信息完整，绕过编码问题，除了可读性差外没有乱码问题。

#### 三、强制程序用 UTF-8 输出

我尝试在 main 函数的开头，使用 `System.IO` 模块的 `hSetEncoding` 函数：

```
main :: IO ()
main = do
  hSetEncoding stdout utf8
  hSetEncoding stderr utf8
```

即显式设置标准输出 (stdout) 和标准错误 (stderr) 的编码为 UTF-8。

这下好了，无论是 `chcp 65001` 还是 `chcp 963` 的终端环境，输出的中文都变成了乱码。

也许还有其他更好的解决方法呢？我暂时只想了这三种可能性，目前看来，还是 JSON 的方法最管用。
