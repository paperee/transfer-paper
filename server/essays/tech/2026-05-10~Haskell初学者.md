### 当我想速成一门编程语言

我会挑选几篇该语言的代码，让 AI 解释里面高频出现的**语法**与**特性**。

这是一种**应用型**的学习方法，我们拿到的，是立马用得上的工具。

（这种感觉跟~~期末前一天晚上刷 B 站速成课~~一样奇妙 ——ee）

适当地**类比**与**对比**先前学过的旧语言与新语言，是再高效不过的。

`claude-haiku-4.5` 评估了这种学习方法，表示**自上而下**有如下优势：

- **快速可用性**：_可以迅速写出能跑的代码，在心理上产生正反馈，激励继续学习；_
- **上下文驱动**：_实战场景帮助塑造"功能导向"的认知网络，而非孤立的语法规则；_
- **语法紧凑**：_代码分析帮助理解语法连招，而非死记硬背"定义变量"等抽象规则。_

但同时，也存在几个隐藏的风险与盲区：

- **样本偏差**：_少数样本没有代表性，我们可能学到某种"特定风格"，而非该语言本身；_
- **忽略边界**：_踩中"边界情况"坑时，在没有知识储备的情况下，调试起来无从下手；_
- **优化盲点**：_"能用"的100行代码不一定"高效"，性能问题或差的设计可能被沿用；_
- **调试黑洞**：_不理解底层机制，遇到 BUG 时，就只能通过低效的"试错"来解决问题。_

因而，在快速入门后，我决定**自下而上**地学习《[Haskell 趣学指南](https://flaneur2020.github.io/)》

#### 惰性求值与 n 阶单位矩阵

尝试利用**递归**与**惰性求值**特性实现该功能，以下是思路（伪代码）：

```haskell
ghci> -- mtx n = [[1*n]*n]
ghci> -- rep ? = ?:rep ?
ghci> -- -> rep 1 = 1:rep 1
ghci> -- -> rep [1,1,…] = [1,1,…]:rep [1,1,…]
ghci> -- mtx n -> take n rep (take n rep 1)
```

简洁的函数，n 表示 n 阶单位矩阵：

```haskell
mtx :: Int -> [[Int]]
-- take 函数用于取无限列表的前 n 项
mtx n = take n (rep (take n (rep 1)))
  -- 将 x 单独提出来而不是包在 rep 中，目的是记录递归节点
  where rep x = x:(rep x)

-- 添加了不少括号用于消歧义；函数式真是又好写又好玩（bushi
```

> Date: 2026.05.05

### 编译时的报错截断

首次尝试使用 `stack built` 编译后，控制台抛出如下报错：

```haskell
Error: [S-7282]
       Stack failed to execute the build plan.
       While executing the build plan, Stack encountered the error:
       <stderr>: commitAndReleaseBuffer: invalid argument (cannot encode character '\65533')
```

报错表示，**标准错误** (stderr) 无法编码 `\65533` 字符。

在 Unicode 中，`\65533` 对应 `U+FFFD` 替换字符。

这个字符没有实际意义，表示数据流中损坏或不兼容的字符。

`chcp` 执行后输出 `活动代码页: 936`，表示终端是 GBK 编码。

`chcp 65001` 切换 UTF-8 编码页，重新编译，出现正常的 GHC 错误信息：

```haskell
app\Main.hs:37:7: error: [GHC-83865]
    鈥?Couldn't match type 鈥楨ither RestCallErrorCode ()鈥?with 鈥?)鈥?[0m
    ……
      In a stmt of a 'do' block:
        when (self `isInfixOf` messageContent msg)
          $ do restCall
                 $ CreateReaction (messageChannelId msg, messageId msg) "fire"
   |
37 |       restCall $ CreateReaction (messageChannelId msg, messageId msg) "fire"
   |       ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
```

首行末出现用于**重置所有文本属性**的 ANSI 转义序列 `[0m`。

而 `鈥` 和 `楨` 等乱码，其实就跟~~锟斤拷~~同个性质。

推测彩色错误信息中，ANSI 序列无法在 GBK 编码下正确翻译。

在当前代码页上运行编译后的可执行文件，却发现**中文乱码**。

尝试切回 GBK 编码，重新运行可执行文件，打印的**中文正常**。

### 终端编码问题

这里有**三条路**可以走：

- 一、_禁用 GHC 的彩色输出_；
- 二、_强制 GHC 用 JSON 输出_；
- 三、_强制程序用 UTF-8 输出_。

#### 一、禁用 GHC 的彩色输出

在 `ghc-options` 中添加 `-fdiagnostics-color=never` 参数，重新编译：

```haskell
C:\sr\setup-exe-cache\x86_64-windows\Cabal-simple_O_vy6YIf_3.12.1.0_ghc-9.10.3.exe --verbose=1 --builddir=.stack-work\dist\1a191874 build lib:hcc-eebot exe:hcc-eebot-exe --ghc-options " -fdiagnostics-color=always"
```

打印的路径和参数表明，实际执行参数还是 `-fdiagnostics-color=always`。

自定义的选项似乎被优先级更高的配置覆盖，或者存在冲突。

使用 `--no-interleaved-output` 参数禁用交错输出模式，同样没用。

#### 二、强制 GHC 用 JSON 输出

在 `ghc-options` 中添加 `-fdiagnostics-as-json`，重新编译：

```haskell
{"version":"1.0","ghcVersion":"ghc-9.10.3","span":{"file":"app\\Main.hs","start":{"line":122,"column":87},"end":{"line":122,"column":88}},"severity":"Error","code":58481,"message":["parse error on input `)'"],"hints":[]}
```

（这个参数使输出为 JSON 纯文本格式，不携带特殊字符）

行号与错误信息完整，成功绕过编码问题，就是可读性有点差。

#### 三、强制程序用 UTF-8 输出

在 main 函数的开头添加 `System.IO` 模块的 `hSetEncoding` 函数：

```haskell
main :: IO ()
main = do
  hSetEncoding stdout utf8
  hSetEncoding stderr utf8
```

即显式设置**标准输出** (stdout) 和**标准错误** (stderr) 的编码为 UTF-8。

这下好了，中文在所有环境下都是乱码。
